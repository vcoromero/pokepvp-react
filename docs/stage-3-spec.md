# Stage 3 — MongoDB and Persistence (Detailed Specification)

This document details what must be built in **Stage 3** of the PokePVP phased plan. It expands on [phased-plan.md](phased-plan.md) and aligns with [architecture.md](architecture.md) and [business-rules.md](business-rules.md). **Status:** ✅ Done.

---

## 1. Goal

Introduce **persistence** using **MongoDB**. The domain will define **repository ports** (output ports) for the entities that need to be stored: Player, Lobby, Team selection, Battle, and Pokémon state. Infrastructure will implement these ports with **MongoDB adapters** (schemas, indexes, and CRUD operations).

**Key principle:** Use cases in Stage 4 (lobby, team, battle) will depend only on these repository abstractions, not on MongoDB. This stage delivers the ports and their MongoDB implementations; verification is done via minimal REST routes or tests that exercise read/write.

---

## 2. Target Folder Structure

Repository **ports** live in `domain/ports/`. **Entities** (or their data shapes) live in `domain/entities/`. MongoDB **adapters** (models/schemas and repository implementations) live under `infrastructure/persistence/`.

```
src/
  index.js
  app.js
  domain/
    ports/
      catalog.port.js           # (existing)
      player.repository.js      # PlayerRepository port
      lobby.repository.js       # LobbyRepository port
      team.repository.js        # TeamRepository port (team selection per player per lobby)
      battle.repository.js      # BattleRepository port
      pokemon-state.repository.js  # PokemonStateRepository port (HP, defeated per battle)
    entities/
      player.entity.js          # Player shape / factory
      lobby.entity.js           # Lobby shape + status enum
      team.entity.js            # Team selection shape
      battle.entity.js          # Battle shape
      pokemon-state.entity.js   # Pokémon state in battle (currentHp, defeated)
  application/
    use-cases/
      # (existing Pokémon use cases)
  infrastructure/
    http/
      catalog.controller.js    # (existing)
      # Optional: minimal persistence.controller.js for Stage 3 verification (e.g. GET/POST lobby, player)
    clients/
      pokeapi.adapter.js       # (existing)
    persistence/
      mongodb/
        connection.js          # MongoDB connection singleton; reads MONGODB_URI
        schemas/
          player.schema.js
          lobby.schema.js
          team.schema.js
          battle.schema.js
          pokemon-state.schema.js
        adapters/
          player.mongo.repository.js
          lobby.mongo.repository.js
          team.mongo.repository.js
          battle.mongo.repository.js
          pokemon-state.mongo.repository.js
    mappers/
      # Optional: domain entity <-> MongoDB document mappers
```

**Naming convention:** Ports are named `XxxRepository` (e.g. `PlayerRepository`). Implementations are named `xxx.mongo.repository.js` (e.g. `player.mongo.repository.js`). Entities define the **domain shape**; schemas define the **storage shape** (with `_id`, timestamps, etc.).

## 3. Entities and Data Shapes

Entities are defined by the domain. They are **plain objects** or **factory functions**; no MongoDB or Mongoose in the domain. The following shapes align with [business-rules.md](business-rules.md) §8.

### 3.1 Player

- **Purpose:** Identity and association to lobby/team.
- **Fields (domain):** `id`, `nickname`, `lobbyId` (optional; set when joined).
- **Note:** `id` can be generated by the repository adapter (e.g. MongoDB `_id` as string).

### 3.2 Lobby

- **Purpose:** Single lobby for one match (2 players).
- **Fields (domain):** `id`, `status`, `playerIds` (array of player ids), `createdAt` (optional).
- **Status:** One of `waiting` | `ready` | `battling` | `finished` (see business-rules §3).

### 3.3 Team (Team selection)

- **Purpose:** Which Pokémon (by catalog id) are assigned to each player for the match.
- **Fields (domain):** `id` (optional), `lobbyId`, `playerId`, `pokemonIds` (array of 3 catalog ids; no duplicates between the two players is enforced in use case, not in schema).

### 3.4 Battle

- **Purpose:** Battle instance linked to a lobby and players.
- **Fields (domain):** `id`, `lobbyId`, `startedAt` (optional), `winnerId` (optional; set when battle ends).

### 3.5 Pokémon state

- **Purpose:** Per-Pokémon state in a battle: current HP and defeated flag.
- **Fields (domain):** `id` (optional), `battleId`, `pokemonId` (catalog id), `playerId`, `currentHp`, `defeated` (boolean).

## 4. Repository Ports (Domain)

Each port is an **interface** (contract). In JavaScript, this is documented (JSDoc) and/or validated at wiring time. Implementations live in infrastructure.

### 4.1 PlayerRepository

**File:** `domain/ports/player.repository.js`

- `save(player)` → `Promise<Player>` — create or update; returns the persisted player (with `id` if new).
- `findById(id)` → `Promise<Player | null>`
- `findByLobbyId(lobbyId)` → `Promise<Player[]>` — players in a given lobby.

### 4.2 LobbyRepository

**File:** `domain/ports/lobby.repository.js`

- `save(lobby)` → `Promise<Lobby>` — create or update; returns the persisted lobby.
- `findById(id)` → `Promise<Lobby | null>`
- `findActive()` → `Promise<Lobby | null>` — single active lobby (e.g. status not `finished`); per business rules, one lobby per match is acceptable.

### 4.3 TeamRepository

**File:** `domain/ports/team.repository.js`

- `save(team)` → `Promise<Team>` — create or update team selection for a player in a lobby.
- `findByLobbyAndPlayer(lobbyId, playerId)` → `Promise<Team | null>`
- `findByLobby(lobbyId)` → `Promise<Team[]>` — all teams in a lobby (both players).

### 4.4 BattleRepository

**File:** `domain/ports/battle.repository.js`

- `save(battle)` → `Promise<Battle>` — create or update.
- `findById(id)` → `Promise<Battle | null>`
- `findByLobbyId(lobbyId)` → `Promise<Battle | null>` — battle for a given lobby.

### 4.5 PokemonStateRepository

**File:** `domain/ports/pokemon-state.repository.js`

- `save(state)` → `Promise<PokemonState>` — create or update one Pokémon state in a battle.
- `findByBattleId(battleId)` → `Promise<PokemonState[]>` — all Pokémon states for a battle.
- `saveMany(states)` → `Promise<PokemonState[]>` — bulk upsert (optional; useful when initializing battle).

## 5. MongoDB Adapters (Infrastructure)

### 5.1 Connection

**File:** `infrastructure/persistence/mongodb/connection.js`

- Read `MONGODB_URI` from environment (required when persistence is used).
- Export a function or singleton to **connect** and return the Mongo client (or Mongoose connection).
- Handle connection errors; do not hardcode the URI.
- **Bootstrap:** `index.js` or `app.js` calls connect before mounting routes that use repositories.

### 5.2 Schemas and Indexes

Use **Mongoose** (recommended) or native MongoDB driver with explicit schema. Define:

- **Player:** `nickname` (string, required), `lobbyId` (ObjectId ref or string). Index: `lobbyId` for `findByLobbyId`.
- **Lobby:** `status` (enum: `waiting`, `ready`, `battling`, `finished`), `playerIds` (array of ObjectId or string), `createdAt`. Index: `status` for `findActive()` (e.g. one document with `status` ne `finished`).
- **Team:** `lobbyId`, `playerId`, `pokemonIds` (array of numbers). Unique compound index: `(lobbyId, playerId)` so one team per player per lobby.
- **Battle:** `lobbyId`, `startedAt`, `winnerId`. Index: `lobbyId` (unique) for `findByLobbyId`.
- **PokemonState:** `battleId`, `pokemonId`, `playerId`, `currentHp`, `defeated`. Compound index: `(battleId, playerId, pokemonId)` for lookups and upserts.

Exact field names (e.g. `_id`, `createdAt`) are part of the adapter; **mappers** in infrastructure can map between domain entities (e.g. `id`) and documents (e.g. `_id`).

### 5.3 Repository Implementations

Each `infrastructure/persistence/mongodb/adapters/*.mongo.repository.js`:

- Implements the corresponding domain port (same method names and return shapes).
- Uses the MongoDB connection and the relevant schema/model.
- Maps documents to domain entities (e.g. `_id` → `id`, strip internal fields).
- Throws or returns in the shape expected by the port (e.g. `null` when not found).

## 6. Configuration and Bootstrap

- **Environment:** Add `MONGODB_URI` to `.env.example` (e.g. `MONGODB_URI=mongodb://localhost:27017/pokepvp`). No default in code; fail fast if persistence is used and `MONGODB_URI` is missing.
- **app.js:** Accept optional `repositories` (or individual repo ports) in `createApp(options)` so tests can inject in-memory or mock repositories.
- **Wiring:** When `MONGODB_URI` is set, create the Mongo connection, create the MongoDB repository adapters, and pass them into the app (e.g. for future use cases or for minimal verification routes).

### 6.1 Running MongoDB: Docker Compose (no backend in Docker)

MongoDB is run via **Docker Compose only**; the backend is **not** containerized in this stage. Evaluators and developers run MongoDB in Docker, then start the backend locally with `npm run dev`.

- **Decision:** One service in `docker-compose.yml`: **mongo** (official image `mongo`, **volume for persistence** `mongo-data:/data/db`). No Dockerfile for the backend.
- **Configurable host port:** The host port is set via **`MONGO_HOST_PORT`** in `.env`. In `docker-compose.yml`, the mapping is `${MONGO_HOST_PORT:-27017}:27017` (default 27017). This lets evaluators who already have MongoDB (or another service) on 27017 use a different port without editing the Compose file: set `MONGO_HOST_PORT=27018` (or any free port) in `.env`, and set `MONGODB_URI=mongodb://localhost:27018/pokepvp` so the app connects to the same port.
- **Usage:** Run `docker-compose up -d` to start MongoDB; then `npm run dev` on the host. Backend connects with `MONGODB_URI=mongodb://localhost:<MONGO_HOST_PORT|27017>/pokepvp`.
- **Documentation:** The README must state: (1) Default: no `MONGO_HOST_PORT` needed; use `MONGODB_URI=mongodb://localhost:27017/pokepvp`. (2) If port 27017 is already in use, set `MONGO_HOST_PORT` to a free port (e.g. 27018) in `.env` and set `MONGODB_URI=mongodb://localhost:27018/pokepvp` (same port) in `.env`.

## 7. Verification in This Stage

Stage 4 will implement lobby/battle **use cases** that depend on these repositories. In Stage 3, verify persistence without full use cases:

- **Option A — Minimal REST routes:** Add a simple controller (e.g. `infrastructure/http/persistence.controller.js`) with:
  - `POST /lobby` — create a lobby (body: `{ status: 'waiting', playerIds: [] }`), return saved lobby.
  - `GET /lobby/active` — get active lobby (calls `LobbyRepository.findActive()`), return JSON.
  - `POST /player` — create a player (body: `{ nickname }`), return saved player.
  - These routes are for **verification only**; they can be removed or restricted in Stage 4 when real use cases and Socket.IO take over.
- **Option B — Tests only:** No new routes; use **integration tests** (Jest + real or in-memory MongoDB) that call the repository adapters directly and assert save/find behavior.

**Implemented:** Option A — `infrastructure/http/persistence.controller.js` with `POST /lobby`, `GET /lobby/active`, `POST /player`. Routes were mounted only when `MONGODB_URI` was set.

**Superseded by Stage 4:** In Stage 3, `POST /player` only accepts `{ nickname }` and does not accept `lobbyId`; it does not update any lobby’s `playerIds`. The domain entity `Player` has an optional `lobbyId` (set when joined), but the verification routes do not use it. Associating a player to a lobby (join lobby) will be implemented in **Stage 4** (use case “Join lobby (nickname)”). As of Stage 4, **LobbyController** at `/lobby` serves the full lobby flow (join, assign-team, ready, get active/by id). PersistenceController is no longer mounted in `app.js`; it remains in the codebase for unit tests. See [stage-4-spec.md](stage-4-spec.md).

## 9. What Changes vs What Stays

| Stage 2 | Stage 3 |
|---------|---------|
| `infrastructure/persistence/` empty | `infrastructure/persistence/mongodb/` with connection, schemas, and repository adapters |
| No domain entities | `domain/entities/` with Player, Lobby, Team, Battle, PokemonState shapes |
| Only CatalogPort | CatalogPort + PlayerRepository, LobbyRepository, TeamRepository, BattleRepository, PokemonStateRepository |
| No database | MongoDB; config via `MONGODB_URI` |
| Bootstrap without DB | Bootstrap connects to MongoDB and wires repository adapters (or injects mocks in tests) |

**Unchanged:**

- All Stage 2 routes and behavior (`/catalog/list`, `/catalog/list/:id`, `/health`).
- Hexagonal structure: domain and application still do not import MongoDB or Express; they depend on repository ports.

## 11. Success Criteria

1. **Ports defined:** All five repository ports exist in `domain/ports/` with the contract described in §4.
2. **Adapters implemented:** Each port has a MongoDB implementation in `infrastructure/persistence/mongodb/adapters/` that respects the contract and maps to domain entities.
3. **Configuration:** MongoDB connection uses `MONGODB_URI`; no hardcoded URLs.
4. **Testability:** `createApp(options)` can receive repository implementations (e.g. mocks) so that Stage 4 use case tests do not require a real database.
5. **Verification:** Either minimal REST routes or integration tests demonstrate that at least one repository can persist and retrieve data correctly.
